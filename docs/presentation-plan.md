# План презентации: MobX-only vs MobX + TanStack Query

## Зачем нужен этот формат

Текущий `walkthrough` полезен как опорный документ для code review-сессии, но он слишком подробный для презентации.
Для самой презентации лучше не идти по всем сценариям подряд, а собрать короткое и понятное повествование:

- показать, что именно сравнивалось
- показать 2-3 репрезентативных фрагмента кода
- объяснить, какую инженерную проблему решает TanStack Query
- честно проговорить trade-offs
- завершить рекомендацией, где какой подход уместен

Главная идея презентации:

`TanStack Query` нужен не потому, что это "более модная библиотека", а потому что он стандартизирует работу с `server state` и снимает повторяемую инфраструктурную сложность.

---

## Цель презентации

Цель выступления:

- не продать новый инструмент любой ценой
- показать результат исследования на одном и том же demo
- дать команде рамку принятия решения для admin screen'ов

Ожидаемый итог для слушателя:

- он понимает, что именно меняется при переходе на `TanStack Query`
- он видит, что `MobX-only` не "плохой", а просто берет на себя больше ручной инфраструктуры
- он понимает, в каких случаях `TanStack Query + MobX` стоит сделать baseline

---

## Рекомендуемый формат

Оптимальный формат презентации:

1. Вступление и рамка сравнения
2. Один слайд про `MobX-only`
3. Один слайд про `TanStack Query + MobX`
4. Три ключевых сценария сравнения
5. Trade-offs и честные минусы
6. Вывод и рекомендация
7. Опционально: very smoke code tour по проекту

Рекомендуемая длительность основной части:

- `10-15 минут` на презентацию
- `5-10 минут` на обсуждение

Важно:

- без live-demo как основной части
- без прохода по всем файлам
- код показывать только как доказательство тезиса, а не как центр всей встречи

## Скелет повествования

## 1. Вступление: что именно сравнивали

Цель блока:

- сразу снять вопрос "а сравнение вообще честное?"

Что сказать:

- в репозитории сделан один и тот же demo-сценарий
- backend, API layer, UI и сценарии одинаковые
- отличается только способ управления состоянием на клиенте
- сравнение не про "hooks vs stores", а про ownership model для `server state`

Ключевая фраза:

"Важно смотреть не на количество строк, а на то, где живет ответственность за server state."

На этом этапе можно показать только 1 слайд-схему:

- `MobX-only`: UI state + server state живут в stores
- `TanStack + MobX`: UI/domain state в MobX, server state в query cache

---

## 2. MobX-only: в чем сила подхода

Цель блока:

- признать реальные плюсы текущего знакомого подхода

Что показать:

- store как явный владелец данных и логики
- container, который вызывает store actions

Какие файлы можно использовать как опорные:

- `src/demo/mobx/stores/counters-store.ts`
- `src/demo/mobx/stores/triggers-store.ts`
- `src/widgets/mobx-dashboard/ui/mobx-dashboard-content.tsx`

Что сказать:

- flow читается линейно
- store API понятен: `fetchCounters`, `fetchCounter`, `removeBan`, `fetchTriggers`, `updateTrigger`
- разработчику легко понять, где лежит логика
- для небольших экранов это может быть достаточно и удобно

Ключевой тезис:

"MobX-only здесь хорош не потому, что он проще технически, а потому что он локально очень явный."

---

## 3. Где MobX-only начинает дорожать

Цель блока:

- мягко перевести разговор от удобства к стоимости поддержки

Что показать на примере `counters-store` и `triggers-store`:

- ручные `loading/fetching/error` флаги
- `AbortController`
- stale request guards
- polling через `setInterval`
- ручная синхронизация list/details
- отдельный cache в `triggersStore`

Что сказать:

- store перестает быть только business/domain owner
- он начинает играть роль самодельного `server-state layer`
- инфраструктурный код начинает расти быстрее, чем бизнес-логика

Ключевая формулировка:

"Проблема не в MobX как таковом. Проблема в том, что вместе с server state мы начинаем вручную писать lifecycle, cache и sync-логику."

Это первый поворотный момент презентации.

---

## 4. TanStack Query + MobX: что именно меняется

Цель блока:

- показать, что речь не про замену MobX целиком

Что показать:

- `MobX` остается для UI state
- `TanStack Query` берет ownership над `server state`

Опорные файлы:

- `src/demo/tanstack/queries/use-counters.ts`
- `src/demo/tanstack/mutations/use-remove-ban.ts`
- `src/demo/tanstack/mutations/use-update-trigger.ts`
- `src/app/providers/query/query-client.ts`
- `src/widgets/tanstack-dashboard/ui/tanstack-dashboard-content.tsx`

Что сказать:

- query hooks описывают чтение данных
- mutation hooks описывают запись и последствия
- общая policy вынесена в `queryClient`
- кэш, retry, staleTime, refetch и deduplication становятся частью стандартного контракта

Ключевой тезис:

"TanStack Query не убирает сложность бизнес-сценариев, но убирает необходимость каждый раз заново собирать инфраструктуру вокруг запросов."

## 5. Три сценария, которых достаточно для презентации

В презентации не нужно проходить все 7 сценариев из walkthrough.
Достаточно трех, потому что они покрывают почти весь смысл перехода.

### Сценарий 1. Загрузка списка

Что сравниваем:

- `MobX`: `store action + flags + useEffect + guards`
- `TanStack`: `useQuery + queryKey + встроенный lifecycle`

Почему этот сценарий важен:

- он самый простой для входа
- на нем уже видно, кто владеет запросом и его жизненным циклом

Главная мысль:

"Даже на простом fetch списка TanStack убирает часть ручного orchestration."

### Сценарий 2. Background refetch / polling

Что сравниваем:

- `MobX`: `setInterval`, cleanup, ручное разделение `isLoading/isFetching`
- `TanStack`: `refetchInterval` как декларативная настройка

Почему этот сценарий важен:

- здесь преимущество TanStack видно особенно отчетливо
- это хороший пример именно инфраструктурной, а не бизнесовой сложности

Главная мысль:

"Polling в MobX быстро превращается в поддержку механики, а не продукта."

### Сценарий 3. Mutation + синхронизация данных

Что сравниваем:

- `MobX`: ручное обновление list/details/cache после изменения
- `TanStack`: `setQueryData` и `invalidateQueries` как единый язык синхронизации

Почему этот сценарий важен:

- он показывает самую дорогую часть server state при росте экранов
- именно здесь хорошо обсуждать `queryKey` и invalidation strategy как trade-off

Главная мысль:

"TanStack не делает sync магическим, но делает его типовым и предсказуемым."

## 6. Какие куски кода реально показывать

Чтобы презентация была лаконичной, код лучше показывать очень дозированно.
Не больше `3-4` фрагментов.

Рекомендуемый набор:

1. `MobX counters store`
   Что показать:
   метод `fetchCounters`, флаги, `AbortController`, polling

2. `TanStack useCounters`
   Что показать:
   `queryKey`, `queryFn`, `refetchInterval`

3. `MobX updateTrigger` или `removeBan`
   Что показать:
   ручную синхронизацию после мутации

4. `TanStack mutation`
   Что показать:
   `setQueryData` или `invalidateQueries`

Дополнительно, если будет нужен один контейнер:

- показать рядом `mobx-dashboard-content.tsx`
- и `tanstack-dashboard-content.tsx`

Но не разбирать их построчно. Только тезисно:

- в `MobX` orchestration больше собран вокруг store actions
- в `TanStack` orchestration распределен между query/mutation hooks и cache policy

## 8. Честные trade-offs, которые нужно проговорить

Этот блок очень важен, потому что он делает вывод убедительным, а не рекламным.

Что нужно признать вслух:

- `MobX-only` локально читается проще и линейнее
- у `TanStack Query` выше входной порог
- нужно понимать `queryKey`, invalidation, cache semantics
- часть сложности действительно "прячется" в библиотеку
- для маленького экрана `MobX-only` может быть рациональнее

Хорошая формулировка:

"Вопрос не в том, есть ли сложность. Вопрос в том, хотим ли мы стандартизировать ее через библиотеку или продолжать реализовывать руками по месту."

---

## 9. Итоговый вывод

Финальный вывод лучше делать не абсолютным, а инженерно-прагматичным.

Рекомендуемая формулировка:

- `TanStack Query + MobX` стоит рассматривать как baseline для admin screen'ов с выраженным `server state`
- `MobX-only` остается нормальным выбором для небольших, локальных и слабо связанных экранов

Фраза для закрытия:

"Если смотреть не на модность, а на стоимость поддержки, TanStack Query выигрывает прежде всего как стандартизированный слой для server state. MobX-only при этом не исчезает, а становится осознанным выбором там, где экран действительно простой."

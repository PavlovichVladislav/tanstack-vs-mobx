# Стратегия управления состоянием для admin panel

## Контекст

Этот demo-проект сравнивает два frontend-подхода для admin panel:

1. TanStack Query + MobX для UI/domain state
2. MobX-only, где server state тоже хранится в store

Цели сравнения:

- одинаковый product scope
- одинаковый UI
- одинаковая декомпозиция компонентов
- без искусственной деградации одной из веток

Контекст команды:

- MobX уже широко используется
- подход MobX-only привычен команде
- store patterns сейчас не стандартизированы
- TanStack Query в команде пока не используется

Цель проекта не в том, чтобы выбрать модный стек, а в том, чтобы понять, оправдан ли новый стандарт для админок с инженерной точки зрения.

## Краткий вывод

Для admin panel такого типа `TanStack Query + MobX` выглядит более сильным кандидатом на роль стандартного подхода для server state.

Это не потому, что MobX-only плохой. Причина в том, что экраны с:

- таблицами
- фильтрами и поиском
- выбранной сущностью и блоком details
- связанными данными
- background refresh
- мутациями
- несколькими потребителями одних и тех же server data

лучше ложатся на специализированный слой для server state.

В текущем командном контексте это преимущество усиливается тем, что MobX stores для server state сейчас пишутся довольно свободно и без единого строгого async/cache pattern.

## Основные выводы

### Где TanStack Query реально снижает сложность

- Shared cache между независимыми потребителями одних и тех же данных
- Единая и понятная модель loading/fetching/error lifecycle
- Встроенные patterns для refetch, invalidation, deduplication и polling
- Меньше ручной синхронизации после mutations
- Ниже риск того, что каждый разработчик будет по-своему реализовывать server-state flows
- Лучше масштабируется по мере роста числа admin screens

### Где выигрыш TanStack ограничен

- Простые CRUD-экраны с одним-двумя запросами
- Экраны без shared consumers одного и того же ресурса
- One-shot loading без сложного refetch/update flow
- Небольшие локальные панели, где explicit stores и так легко читать

В таких случаях TanStack может быть лучше, но не драматически лучше.

### Где MobX-only может быть лучше

- Небольшие и локальные экраны с простым data flow
- Сценарии, где явный orchestration в одном месте читается для команды проще
- Команды, которым ближе модель stores/actions, чем query cache semantics
- Ситуации, где организационная стоимость нового инструмента слишком высока

MobX-only особенно силен в локальной явности.
TanStack Query особенно силен в системном управлении server state.

## Организационная оценка

У TanStack Query есть реальная стоимость внедрения:

- query keys
- invalidation strategy
- cache updates после mutations
- enabled/refetch semantics
- новая mental model: server state отдельно от UI state

Но у MobX-only в текущем командном контексте тоже есть постоянная стоимость:

- ручная работа с lifecycle
- ручная логика polling/refetch
- ручная синхронизация после mutations
- непоследовательные store patterns между разработчиками

Так как сейчас внутри команды нет сильного внутреннего стандарта для MobX async/server state, TanStack Query может решать не только техническую задачу, но и задачу стандартизации.

Learning cost оправдан, если:

- команда действительно хочет единый подход для server state в админках
- data-heavy экранов много
- команда готова договориться о conventions и закрепить их

Learning cost не оправдан, если:

- большинство экранов на практике сильно проще этого demo
- команда не готова вкладываться в conventions и onboarding
- результатом станет смешанная и неуправляемая архитектура

## Варианты стратегии

### Strategy A - Остаться на MobX-only

Плюсы:

- нет стоимости внедрения нового инструмента
- команда уже знакома с этим подходом
- explicit orchestration легко отслеживать
- для простых экранов подход вполне достаточный

Минусы:

- server-state инфраструктура остается ручной
- высокий риск непоследовательных реализаций
- cache/refetch/polling/mutation sync остаются на усмотрение разработчика
- стоимость поддержки растет вместе со сложностью экранов

Когда оправдано:

- если большинство admin screens простые
- если команда не готова инвестировать в новый стандарт
- если вместо этого команда готова жестко формализовать MobX async-store pattern

### Strategy B - Перейти на TanStack Query + MobX

Плюсы:

- более сильный baseline для server state
- меньше повторяющейся инфраструктурной логики
- более предсказуемый lifecycle и data flow
- лучше подходит для shared data, refetch, polling и mutations
- лучше решает задачу стандартизации

Минусы:

- есть learning curve
- нужна дисциплина вокруг query keys и invalidation
- в frontend stack появляется второй state tool
- для некоторых простых экранов это может быть избыточно

Когда оправдано:

- если админки data-heavy
- если много экранов переиспользуют или обновляют одни и те же данные
- если команда хочет именно стандарт, а не просто дополнительную опцию

### Strategy C - Разрешить оба подхода

Плюсы:

- максимальная гибкость
- подход можно выбирать под конкретный экран
- организационно это мягче, чем полный переход

Минусы:

- самый высокий риск архитектурной фрагментации
- постоянные споры, какой подход выбирать
- onboarding и review становятся сложнее
- без строгих критериев стандарт по факту не появится

Когда оправдано:

- если команда может четко описать критерии выбора
- если оба подхода уже хорошо понятны
- если гибкость важнее стандартизации

## Итоговая рекомендация

Рекомендуемый подход:

- `TanStack Query + MobX для UI/domain state`

Почему:

- Для admin panel такого типа TanStack Query действительно уменьшает объем повторяющейся инфраструктуры вокруг server state
- В вашем командном контексте главная ценность не только в техническом удобстве, но и в стандартизации
- MobX при этом остается уместным и сильным инструментом для UI и domain state

Когда эта рекомендация может быть неверной:

- если большинство реальных admin screens заметно проще, чем это demo
- если команда не готова инвестировать в conventions и onboarding
- если внутри команды будет создан и реально поддержан сильный MobX-only стандарт для async/server state

## Практическое замечание по rollout

Если команда принимает эту рекомендацию, rollout не должен выглядеть как "теперь TanStack можно использовать где угодно".

Он должен выглядеть так:

- TanStack Query для server state в admin panels
- MobX для UI/domain state
- team conventions для query keys, invalidation, mutations, polling и error handling
- MobX-only допустим только как исключение для маленьких локальных экранов
